# Developing and Deploying Sock Shop

All services are always built from source and deployed as dev images. There
are no stock/upstream image deployments — every deployment uses locally built code.

## What's in this repo

| Directory | Contents |
|-----------|----------|
| `deploy/kubernetes/manifests/` | Base Kustomize manifests (all services) |
| `deploy/kubernetes/overlays/kind/` | Kustomize overlay for kind (generated by `bin/deploy`) |
| `deploy/kubernetes/overlays/eks/` | Kustomize overlay for EKS (generated by `bin/deploy`) |
| `deploy/kubernetes/overlays/gke/` | Kustomize overlay for GKE (generated by `bin/deploy`) |

## Image tagging

Images are tagged with the git SHA of the commit they were built from:

| Scenario | Tag format | Example |
|----------|-----------|---------|
| Clean working tree | `dev-<sha>` | `dev-a3f7c21` |
| Uncommitted changes | `dev-<sha>-dirty` | `dev-a3f7c21-dirty` |

Each service tracks its own tag independently. If you change only `orders/`,
a rebuild will produce a new tag for orders while other services keep their
previous tags. This means the cluster can have services at different SHAs —
each pinned to the exact code it was built from.

## Incremental builds

`bin/build` automatically skips services whose source code hasn't changed
since the last build. It does this by:

1. Reading `.build-manifest.json` (written by the previous build)
2. For each service, running `git diff --quiet <last-built-sha> HEAD -- <service-dir>/`
3. Skipping services with no changes; rebuilding only those that changed

Use `--force` to rebuild everything regardless of changes.

**When dirty builds are forced:** if the working tree has uncommitted changes
(`-dirty` tag), all services are rebuilt because git diff can't reliably detect
which service directories are affected by unstaged changes.

## Build and deploy

Both `bin/build` and `bin/deploy` require a target flag: `--kind`, `--eks`, or `--gke`.

### kind (local development)

```bash
./bin/build --kind              # build changed services, load into kind
./bin/build --kind --force      # rebuild all services
./bin/deploy --kind             # deploy
```

### EKS

```bash
./bin/build --eks               # build changed services for amd64, push to ECR
./bin/build --eks --force       # rebuild all services
./bin/deploy --eks              # deploy
```

### GKE

```bash
./bin/build --gke               # build changed services for amd64, push to Artifact Registry
./bin/build --gke --force       # rebuild all services
./bin/deploy --gke              # deploy
```

### Build then deploy in one step

```bash
./bin/build --kind && ./bin/deploy --kind
./bin/build --eks && ./bin/deploy --eks
./bin/build --gke && ./bin/deploy --gke
```

## Build manifest

`bin/build` writes `.build-manifest.json` (git-ignored) after each run:

```json
{
  "target": "eks",
  "git_sha": "a3f7c21",
  "dirty": false,
  "tag": "dev-a3f7c21",
  "built_at": "2026-03-04T20:15:00Z",
  "services": {
    "carts":    { "tag": "dev-e8b2f10", "image": "…/sockshop/carts:dev-e8b2f10",    "source_sha": "e8b2f10" },
    "orders":   { "tag": "dev-a3f7c21", "image": "…/sockshop/orders:dev-a3f7c21",   "source_sha": "a3f7c21" },
    ...
  }
}
```

`bin/deploy` reads this manifest to:
1. Generate the kustomization overlay with per-service image tags
2. Validate the target matches (errors if you `build --kind` then `deploy --eks`)

## Tracing a deployed image back to code

```bash
# See what's running
kubectl get deploy -n sock-shop -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.template.spec.containers[0].image}{"\n"}{end}'

# Image tag contains the SHA — inspect the code
git log dev-a3f7c21    # or: git log a3f7c21
git diff e8b2f10..a3f7c21 -- orders/
```

## Building a single service

After a full build, you can rebuild just one service manually. The next
`bin/build` will detect the change and skip the service if you've already
rebuilt it.

### Java services (carts, orders, shipping, queue-master)

**kind:**
```bash
docker run --rm -v "$PWD/orders":/build -w /build \
  maven:3-eclipse-temurin-17 mvn package -DskipTests -q
docker build -t weaveworksdemos/orders:dev-$(git rev-parse --short=7 HEAD) orders/
kind load docker-image weaveworksdemos/orders:dev-$(git rev-parse --short=7 HEAD) --name tokonoma
kubectl rollout restart deployment/orders -n sock-shop
```

**EKS:**
```bash
SHA=$(git rev-parse --short=7 HEAD)
REGISTRY=853962316687.dkr.ecr.us-west-2.amazonaws.com

docker run --rm --platform linux/amd64 -v "$PWD/orders":/build -w /build \
  maven:3-eclipse-temurin-17 mvn package -DskipTests -q
docker build --platform linux/amd64 -t $REGISTRY/sockshop/orders:dev-$SHA orders/
docker push $REGISTRY/sockshop/orders:dev-$SHA
kubectl rollout restart deployment/orders -n sock-shop
```

### Go services (catalogue, payment, user)

```bash
SHA=$(git rev-parse --short=7 HEAD)
docker build -t weaveworksdemos/catalogue:dev-$SHA -f catalogue/docker/catalogue/Dockerfile catalogue/
kind load docker-image weaveworksdemos/catalogue:dev-$SHA --name tokonoma
kubectl rollout restart deployment/catalogue -n sock-shop
```

### Node.js (front-end)

```bash
SHA=$(git rev-parse --short=7 HEAD)
docker build -t weaveworksdemos/front-end:dev-$SHA front-end/
kind load docker-image weaveworksdemos/front-end:dev-$SHA --name tokonoma
kubectl rollout restart deployment/front-end -n sock-shop
```

## Kustomize layout

```
deploy/kubernetes/
  manifests/                    Base Kustomize resources (upstream image refs as defaults)
    kustomization.yaml          Lists all manifest files
    00-sock-shop-ns.yaml
    01-carts-dep.yaml
    ...
  overlays/
    kind/kustomization.yaml     Generated by bin/deploy from .build-manifest.json
    eks/kustomization.yaml      Generated by bin/deploy from .build-manifest.json
    gke/kustomization.yaml      Generated by bin/deploy from .build-manifest.json
```

The overlay `kustomization.yaml` files are generated at deploy time with
per-service image overrides derived from the build manifest. The base manifests
use upstream image tags as Kustomize defaults; they are never deployed
directly — all deployments go through an overlay.
